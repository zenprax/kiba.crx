HYOK / BYOE

所有と保管の完全分離

Hold Your Own Key（HYOK）/ Bring Your Own Encryption（BYOE）の原則を徹底実装。 クラウドは「中身を1ミリも解読できない暗号箱の預かり所」に機能を縮退させ、 データ主権を100%顧客エッジに帰属させる。

Opaque Repository

不透過リポジトリとしてのクラウド

サーバーはポリシーBlobと暗号化済みキー（wrappedKey）を 機械的に保管・転送するだけ。クエリ処理・再計算・スキーマ解釈能力を一切持たない 完全ステートレスリポジトリとして設計。

Edge Sovereignty

エッジ側ブラウザの完全自律

復号・ポリシー解決・属性評価はすべてブラウザ拡張機能のローカルメモリ内で完結。 compileActiveSettings() は クラウド接続なしに自律実行し、結果をメモリに保持する。

Cryptographic Zero Disclosure

開示要求への暗号学的無力化

捜査令状・内部不正・ハッキングによるサーバー完全制圧が起きても、 物理的に K_master が存在しないため、 どれだけ圧力をかけても機密を開示する手段が数学的に存在しない。

§3 — Dual-Layer Envelope Encryption Protocol

なぜ2重構造が必要か

1つの鍵で直接暗号化する「シングルキー方式」には、鍵ローテーションと暗号疲弊という 2つの解決不能な問題がある。エンベロープ暗号はこれを構造で解決する。

暗号学的疲弊（Key Fatigue）の完全回避

▶ 同一鍵・大量暗号化の数学的脆弱性

AES-GCM では同じ鍵で大量のデータを暗号化すると、 nonce衝突確率が 2^32 ブロックで無視できなくなる。 解析攻撃でパターンが統計的に漏洩するリスクがある。

▶ ランダムDEKによる完全防衛
データ生成ごとに crypto.getRandomValues() で K_data を生成
各 K_data は1回だけ使用して即時廃棄（use-once key）
K_master は鍵ラッピングのみに使用、大量データに触れない
// crypto.ts — DEK生成（実際のコード）
const kData = crypto.getRandomValues(
  new Uint8Array(32)  // 256-bit DEK
);
// ... 暗号化処理 ...
kData.fill(0);  // 即時ゼロフィル
